     Port tic-tac-toe to vanilla JS with a clickable board UI

     Context

     The project currently has a Python CLI tic-tac-toe at tic_tac_toe.py with unit tests at test_tic_tac_toe.py. The CLI flow (render text board → prompt for 1–9 →
     re-render) makes a richer UI awkward. We're moving to vanilla browser JS so the player can click cells directly on a visible board.

     The first UI iteration is an "open board UI": the 3×3 grid is always shown, the current player clicks the cell where they want to place their shape, and empty cells
     are visually distinct (clear borders, hover state, distinct background) so the player can tell exactly where each space is. Shapes are drawn as inline SVG (red X,
     dark-blue O), not text.

     Game logic stays in a separate module so it can be unit-tested under Node's built-in test runner, mirroring the existing Python test coverage 1:1.

     Decisions

     - Stack: vanilla HTML/CSS/JS, no build step. Open index.html directly in a browser (file://).
     - Module strategy: .js files everywhere with a tiny package.json containing "type": "module". Imports use explicit .js extensions so the same file works in both
     browser and node --test.
     - Cells: <button> elements (free keyboard access + disabled state for played cells).
     - Validation reuse: UI calls parseMove(String(idx + 1), board) so clicks go through the same validator the tests cover.
     - Shapes: inline SVG strings produced by a helper in ui.js.
     - Tests: node:test + node:assert/strict, run via node --test tests/game.test.js.
     - Python files are deleted as part of this work.

     src/ui.js — DOM wiring

     Imports EMPTY, newBoard, applyMove, checkWinner, isFull, parseMove, playerForTurn from './game.js'.

     Module state: board, turn, gameOver.

     Functions:
     - buildCells() — runs once at startup. Creates 9 <button class="cell" data-index="N"> children of #board. Attaches a single delegated click listener that resolves the
     index from e.target.closest('.cell').
     - handleClick(idx):
       a. Return if gameOver.
       b. try { parseMove(String(idx + 1), board); } catch { return; } — same validator the tests cover (rejects occupied cells).
       c. const { num, mark } = playerForTurn(turn); applyMove(board, idx, mark); turn++;
       d. Check terminal state: winner → setStatus(Player ${num} (${mark}) wins); else isFull → setStatus('Tie'); else compute next player and announce their turn.
       e. renderBoard().
     - renderBoard() — for each cell: clear innerHTML; if filled, set inner SVG and disabled = true; if empty and not over, disabled = false; if empty and game over,
     disabled = true.
     - setStatus(text) — sets statusEl.textContent.
     - resetGame() — board = newBoard(); turn = 0; gameOver = false; set status to "Player 1 (X) — your turn", call renderBoard().
     - svgFor(mark) — returns inline SVG string:

     function svgFor(mark) {
       if (mark === 'X') return `
         <svg viewBox="0 0 100 100" xmlns="http://www.w3.org/2000/svg" aria-label="X">
           <line x1="20" y1="20" x2="80" y2="80" stroke="#c0392b" stroke-width="12" stroke-linecap="round"/>
           <line x1="80" y1="20" x2="20" y2="80" stroke="#c0392b" stroke-width="12" stroke-linecap="round"/>
         </svg>`;
       if (mark === 'O') return `
         <svg viewBox="0 0 100 100" xmlns="http://www.w3.org/2000/svg" aria-label="O">
           <circle cx="50" cy="50" r="32" fill="none" stroke="#2c3e50" stroke-width="12"/>
         </svg>`;
       return '';
     }

     Bootstrap at bottom: buildCells(); resetEl.addEventListener('click', resetGame); resetGame();
