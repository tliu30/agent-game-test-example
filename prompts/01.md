     Plan: CLI Tic-Tac-Toe Game

     Context

     The repo (/home/anthony/Projects/agent-game) is essentially empty — only README.md and SPEC.md exist. SPEC.md asks for a single-computer, two-player CLI tic-tac-toe
     game written in Python using only the standard library, with | and - for the board, numbered positions for input, win/tie detection, and a clean exit.

     The spec leaves a few things implicit; clarified with the user:
     - Board layout: row-major (top-left = 1, bottom-right = 9).
     - Win message: Player 1 (X) wins / Player 2 (O) wins.
     - Invalid input (non-numeric, out of range, or already occupied): print an error and reprompt the same player.
     - Include unit tests using stdlib unittest.
     - Build with red-green TDD: write the failing tests first, then add implementation until they pass.

     Goal: produce a small, readable, dependency-free script plus a focused test file, developed test-first.

     Files to create

     - tic_tac_toe.py — the game (single file, runnable as python3 tic_tac_toe.py).
     - test_tic_tac_toe.py — unit tests for board logic, runnable as python3 -m unittest.

     No new directories needed; both go at the repo root next to SPEC.md.

     Design

     Keep it a single module with a tiny set of pure functions for game logic plus a thin main() that does I/O. This makes the logic trivial to unit test without mocking
     input()/print() extensively.

     Data model

     - Board: a list[str] of length 9. Each cell is "X", "O", or " " (space) for empty.
     - Indices 0–8 internally; user-facing positions are 1–9. Convert at the I/O boundary.

     Pure functions (testable)

     - new_board() -> list[str] — returns 9 spaces.
     - render(board) -> str — returns the multi-line board string. Empty cells display their position number (1–9) so the player always sees what to type; filled cells show
      X or O. Uses | between cells and a row of - between rows, per spec.
     - parse_move(raw, board) -> int — strips input, validates it is a digit in 1–9, converts to 0-based index, and verifies the cell is empty. Raises ValueError with a
     human-readable message on any failure. Centralizing this lets tests cover all the invalid-input branches.
     - apply_move(board, idx, mark) -> None — writes mark into board[idx] (mutates).
     - WIN_LINES — module-level constant: the 8 winning index triples (3 rows, 3 cols, 2 diagonals).
     - check_winner(board) -> str | None — returns "X", "O", or None by scanning WIN_LINES.
     - is_full(board) -> bool — true when no empty cells remain.

     I/O layer

     - prompt_move(player_num, mark, board) -> int
       - Prints the rendered board.
       - Loops: reads input(f"Player {player_num} ({mark}), enter position (1-9): "), calls parse_move, prints the error and re-loops on ValueError, otherwise returns the
     index.
     - main()
       - board = new_board()
       - Players cycle: [(1, "X"), (2, "O")]. Loop up to 9 turns.
       - Each turn: get index from prompt_move, apply_move, then check check_winner. If a winner, print the final board and Player {n} ({mark}) wins, then return. Otherwise
      if is_full, print board and Tie, then return.
     - if __name__ == "__main__": main() guard so the test file can import without side effects.

     Rendering detail

     render produces something like:

      1 | 2 | 3
     -----------
      4 | X | 6
     -----------
      7 | 8 | O

     Empty cells show their 1-based position (so the prompt is self-documenting); filled cells show the mark. Width is fixed at 11 characters per row so the - separator
     lines up.

     Tests (test_tic_tac_toe.py)

     Use unittest.TestCase. Cover the pure logic only — no input() mocking needed.

     - test_new_board_is_empty — 9 spaces.
     - test_render_shows_position_numbers_when_empty — fresh board contains "1" and "9".
     - test_render_shows_marks — after placing X at 5, rendered output contains "X" and not "5".
     - test_parse_move_valid — "1" → 0, "9" → 8, surrounding whitespace tolerated.
     - test_parse_move_non_numeric — "a" raises ValueError.
     - test_parse_move_out_of_range — "0" and "10" raise.
     - test_parse_move_occupied — placing X then parsing the same square raises.
     - test_check_winner_row / _column / _diagonal — one case each.
     - test_check_winner_none — empty and mixed-no-winner boards return None.
     - test_is_full — false when empty, true when fully filled.

     Run with python3 -m unittest from the repo root.

     Implementation order (TDD)

     1. Red — write failing tests. Create test_tic_tac_toe.py with all the cases listed above. Create tic_tac_toe.py containing only the stubs the tests import (new_board,
     render, parse_move, apply_move, check_winner, is_full, WIN_LINES) so imports resolve. Stubs should raise NotImplementedError (or return obviously-wrong values). Run
     python3 -m unittest and confirm the suite fails — every logic test should be red for the right reason (assertion / NotImplementedError, not ImportError).
     2. Green — implement pure functions. Fill in new_board, WIN_LINES, check_winner, is_full, apply_move, then parse_move, then render. After each function, rerun python3
     -m unittest; the relevant tests should turn green incrementally. Stop when the whole suite passes.
     3. Add the I/O layer. Implement prompt_move and main() plus the if __name__ == "__main__" guard. These are not unit-tested; verify them by hand in the next step.
     4. Refactor pass. Re-read both files for clarity and dead code; tests must stay green.

     Verification

     1. python3 -m unittest — all tests pass.
     2. python3 tic_tac_toe.py — manually play through three scenarios:
       - X wins (e.g., 1, 4, 2, 5, 3 → row win) → prints Player 1 (X) wins.
       - O wins (force a column or diagonal) → prints Player 2 (O) wins.
       - Tie game (fill all 9 with no line) → prints Tie.
     3. Manually exercise invalid input during a play: type foo, 0, 10, and an already-taken square — each should print a clear error and reprompt the same player without
     advancing the turn.
     4. Confirm the program exits cleanly (returns from main) after each end state.
